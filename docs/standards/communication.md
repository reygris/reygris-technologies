# Communication Standards

Clear communication is a ReyGris product feature. These standards apply internally and with clients.

## Writing

### General rules
- Lead with the conclusion or request, then context
- Use short paragraphs and scannable headings
- Prefer active voice: "We will deploy Tuesday" not "Deployment will be performed"
- Define acronyms on first use in external docs
- Proofread — typos erode trust in a standards-driven company

### Email & async messages
- Subject lines state intent: `[ReyGris] Proposal follow-up — timeline question`
- One primary ask per message when possible
- Link to docs instead of duplicating long explanations

### Documentation
- Markdown in this repo is the source of truth for company standards
- Project docs live with the project; link back to ReyGris standards where relevant
- Update docs when behavior changes — stale docs are worse than none

## Meetings

| Rule | Rationale |
|------|-----------|
| Default to async | Respect focus time |
| Agenda required | No agenda, no meeting |
| Time-box | End early when done |
| Notes & actions | Every decision meeting produces written output |
| Optional attendance | Invite only those who must decide or implement |

## Client communication

- Set expectations early: scope, timeline, assumptions, risks
- Report status proactively — clients should never have to chase
- Escalate blockers within one business day
- Use plain language; explain tradeoffs honestly
- Never over-promise to win work

## Feedback

- Specific, timely, and kind
- Focus on behavior and outcomes, not character
- Offer suggestions when raising problems
- Receive feedback with curiosity, not defensiveness

## Brand voice in public

Follow [Brand Guidelines](../brand.md):

- Confident, direct, warm, precise
- Tagline: *Built with vigilance. Rooted in home.*
- Avoid hype and empty superlatives

## Crisis & incident communication

When something goes wrong in production or with a client:

1. Acknowledge quickly
2. State what is known and unknown
3. Describe immediate mitigation
4. Follow up with root cause and prevention

Blameless postmortems internally; accountable, clear updates externally.
